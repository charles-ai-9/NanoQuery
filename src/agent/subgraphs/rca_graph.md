# 🕵️‍♂️ RCA (Root Cause Analysis) 重案组：根因分析深度指南

## 1. 什么是 RCA？
**RCA** 全称是 **Root Cause Analysis**（根本原因分析）。
在金融风控领域，RCA 的任务不是简单地告诉你“发生了什么”，而是挖掘“为什么会发生”。

* **普通查询 (Business Node)**：上个月有多少人逾期？ -> `500人`。
* **RCA 分析 (RCA Subgraph)**：为什么上个月逾期人数环比增长了 20%？ -> `因为某地推广渠道引入了大量低分段用户，该群体的逾期率高达 15%`。

---

## 2. 为什么需要子图 (Subgraph)？

当调查过程变得极其复杂时，我们需要子图：
1. **职责分离**：主图负责“接待和分流”，子图负责“深度垂直分析”。
2. **逻辑循环**：RCA 往往需要【假设 -> 查证 -> 否定 -> 再假设 -> 再查证】的多轮循环。
   - 第一步（提出假设）：是不是某批新用户的信用分特别低？或者是不是某个特定的贷款产品出了问题？
   - 第二步（多轮查证）：写 5 个不同的 SQL 分别去查不同维度的数据，进行交叉比对。
   - 第三步（剥丝抽茧）：发现原来是上个月业务部门搞了个“无门槛拉新”活动，进来了大量 credit_score 低于 400 分的用户，这批人导致了坏账飙升。
   - 第四步（结案陈词）：给老板写一份详细的《逾期飙升根因分析报告》。
3. **记忆隔离**：子图中间反复查询的大量零碎数据（草稿纸）不需要展示给终端用户，只把最后的“结案报告”发回去。

---

## 3. RCA 办案流程图解


RCA 的工作逻辑通常遵循以下四个阶段：

### 第一阶段：现状扫描 (Fact Finding)
AI 会先查询基础汇总数据，确认异常确实存在。
> SQL: `SELECT status, count(*) FROM repayment_record GROUP BY status`

### 第二阶段：多维拆解 (Multidimensional Drill-down)
AI 会从不同维度去“切片”观察数据。
* **时间维度**：是某一天突然暴增，还是持续增长？
* **人群维度**：是新用户有问题，还是老用户有问题？
* **产品维度**：是信用贷有问题，还是抵押贷有问题？

### 第三阶段：假设验证 (Hypothesis Testing)
这是 RCA 最聪明的地方。它会生成猜想：
> “我怀疑是新用户的信用评分下调导致了风险。”
然后它会去写 SQL 验证这个猜想：
> SQL: `SELECT credit_score, overdue_rate FROM users JOIN loans ...`

### 第四阶段：归因总结 (Final Attribution)
把所有证据串联起来，给出最终结论。

---

## 4. 在代码中如何体现？

在 `rca_graph.py` 中，你通常会看到以下结构：

1. **RCA State**：比主图多出 `hypotheses`（假设列表）和 `evidence`（证据库）字段。
2. **Analysis Node**：大脑不再是写单条 SQL，而是规划“调查路线图”。
3. **Summary Node**：负责把一大堆 SQL 结果翻译成人类能看懂的分析报告。

---

## 5. 学习建议：如何进化你的侦探？

当你阅读 `rca_graph.py` 代码时，重点观察：
* **它是如何决定什么时候结束调查的？** (Stop Condition)
* **它是如何处理 SQL 报错的？** (Error Handling)
* **它是如何在没有直接答案时进行“推理”的？** (Inference)

# Q&A

#### 1. 在咱们的代码里，rca_graph.py 是怎么工作的？
当你把一个 Agent 升级为 RCA 架构时，它通常会在内部进行 “假设 -> 验证 -> 总结” 的循环：

- 脑洞大开：AI 会先思考导致数据异常的几种可能原因（比如：渠道问题、人群问题、系统 Bug）。

- 疯狂调工具：它会在内部连续调用好几次 execute_sql 工具，分别去验证它刚才猜想的原因。

- 拼凑真相：它把所有查到的零碎数据汇总起来，告诉你最根本的原因是什么。

这也是为什么在复杂的企业级 AI 项目里，我们需要单独开一个 rca_graph.py（子图）来处理这种任务。因为这种分析过程太长、太复杂了，如果全塞在前台接待处（graph.py），前台大厅就乱套了。

简单来说：RCA 就是从“读数据的机器”进化成“懂业务的分析师”的关键一步！
