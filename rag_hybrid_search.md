# 🚀 架构提案：NanoQuery 引入混合检索（Hybrid Search）的深度分析

## 1. 核心背景：金融风控场景的特殊性
在“金融风控侦探社”中，Agent 处理的不是普通的闲聊，而是极度依赖**精准度**和**行业术语**的专业对话。
金融知识库通常由两部分组成：
* **语义逻辑**：如“催收政策的执行原则”、“风险评估的指导思想”。
* **硬核术语**：如 `DPD30`、`M1`、`LTV`、`征信白名单`、特定的错误代码 `ERR-404-SQL`。

## 2. 为什么“纯向量检索”在本项目中不够用？
虽然目前的向量检索（FAISS）具备语义理解能力，但在以下关键金融场景中容易出现“识别漂移”：

### 场景 A：首字母缩写与行业黑话
* **用户输入**：💬 “帮我查一下 **DPD** 的定义。”
* **向量检索痛点**：在向量空间中，“DPD”作为简写，其语义向量可能较单薄。若手册中缺乏丰富的上下文解释，模型容易发生误判。
* **混合检索优势**：**关键词检索 (BM25)** 采用字面匹配，能够百分之百锁定“DPD”字符。

### 场景 B：特定的操作禁令（高风险边界）
* **用户输入**：💬 “针对 **M1** 客户，能不能拨打电话？”
* **向量检索痛点**：由于“M1”与“M2”语义接近，向量距离极短。检索可能错误地带回 M2 的政策（允许拨号），导致合规事故。
* **混合检索优势**：通过关键词权重，确保“M1”这一精确标签被优先识别。

## 3. 混合检索方案设计：双路归一化
系统将采用 **“多路召回 + 互惠排名融合 (RRF)”** 的进阶架构：

| 维度 | 方案 1：向量检索 (Dense) | 方案 2：关键词检索 (Sparse) |
| :--- | :--- | :--- |
| **底层技术** | FAISS + HuggingFace Embedding | Rank-BM25 算法 |
| **擅长领域** | “意思相近”的模糊查询（查语义） | “一字不差”的术语匹配（查字面） |
| **角色比喻** | 像一个**博学但马虎**的教授 | 像一个**死板但精准**的校对员 |

## 4. 业务价值总结
* **零幻觉决策**：引用手册原文，而非 LLM 的逻辑推测。
* **专业度跃迁**：精准对齐资深风控专家的行业认知。
* **系统鲁棒性**：即便输入极简，也能通过关键词索引快速定位。
